# Purpose
This skill helps Codex create grounded, persuasive interview answers based on the user's real background and current positioning.

It is especially useful when the user needs to explain:
- career transition
- independent project work
- family business period
- gap-related concerns
- why field-sales experience connects to planning / SFE / CRM roles

# Main interview identity
The user is a senior field-experienced operator who is moving from performance execution into performance system design.

The user should sound:
- practical
- mature
- structured
- credible
- business-oriented

The user should not sound:
- defensive
- vague
- overhyped
- purely technical
- like someone trying to pretend they already held an official planning title

# Core answer logic
Use this answer logic in most interview responses:

1. What I actually did
2. What I learned from that experience
3. Why that led me toward planning / system work
4. What I built or researched to validate that direction
5. Why that makes me relevant to this role now

# Key interview themes

## 1. Why the user fits planning / SFE / CRM roles
Main logic:
The user spent 14 years in the field and saw repeatedly that performance differences were not created by effort alone, but by targeting quality, execution rhythm, portfolio strategy, and management structure.

That insight led the user to focus on:
- KPI logic
- reporting structure
- coaching standards
- activity-to-result linkage
- practical sales operating models

## 2. How to explain the family business period
Recommended framing:
The family business period was practical business operation experience where the user handled marketing, customer management, and operations directly.
It reinforced the importance of operational standards and data-based management in small, inefficient environments.

Do not frame it apologetically.
Do not call it meaningless downtime.

## 3. How to explain the learning / development period
Recommended framing:
The user intentionally strengthened capability in data-based decision making and verified that learning through working prototype projects.

The point is not to claim formal tech seniority.
The point is to show deliberate capability building tied to business problems.

## 4. How to explain the independent project period
Recommended framing:
The user did not stop at studying concepts.
The user structured real sales-management problems into operating logic and built prototypes to test whether the logic could work in practice.

This period proves:
- problem ownership
- planning ability
- structured thinking
- execution discipline
- business-tool design capability

# How to answer likely concerns

## Concern: "You were in sales, not planning."
Recommended logic:
That is true in title, but the user's field experience exposed the actual operating problems that planning functions are supposed to solve.
The user's transition is built on converting repeated field pain points into KPI logic, reporting standards, and management structures.

## Concern: "You have a gap."
Recommended logic:
Do not deny the timeline.
Explain each phase with purpose:
- family business operations
- capability development
- independent project validation

Present the period as a progression, not as a blank.

## Concern: "Is Sales Data OS a real product?"
Recommended logic:
Do not oversell it as a full market product.
Say it is an independently designed and validated prototype / operating framework built to test practical sales-management logic and reporting structure.

## Concern: "Are you more of a developer now?"
Recommended logic:
No.
The user's technical work is a means to validate business operating logic, not the primary identity.
The user's main value is designing management structures that field teams and managers can actually use.

# Preferred answer structure
Unless the user asks otherwise, build answers in this order:
1. direct answer
2. supporting background
3. concrete example
4. implication for the target role

# Tone rules
- calm
- direct
- confident
- non-defensive
- no self-pity
- no exaggerated confidence

# Good framing examples
Use patterns like:
- I started from field execution, but repeatedly saw the limits of result-only management.
- Over time, I became more interested in the structure behind performance.
- That led me to build a more management-oriented perspective around KPI, reporting, and execution standards.
- My recent project work was a way to validate that operating logic in a concrete form.
- I see this role as a place where my field understanding and operating design perspective can be used together.

# Avoid
Avoid:
- trying to sound like a software engineer
- pretending to have held official planning titles that did not exist
- claiming product-market validation that is not real
- making the answer too long before stating the point
- overexplaining the gap emotionally

# Output patterns
## Pattern A: 1-minute self-introduction
- background

- role relevance

## Pattern B: transition explanation
- field foundation
- insight gained
- capability building
- current direction

## Pattern C: role-fit answer
- why the role matches past experience
- what current project work adds
- what the user can contribute immediately

# Special caution
When writing interview answers:
- make the user sound senior and grounded
- always connect experience to business use
- keep Sales Data OS as evidence of structured thinking, not as hype
- preserve the user's identity as a field-informed planning / SFE / CRM candidate
