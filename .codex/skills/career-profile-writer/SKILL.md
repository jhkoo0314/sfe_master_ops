# Purpose
This skill helps Codex explain who the user is, what the user has done, and how the user's career should be framed in clear business language.

It should be used whenever the output depends on the user's real career history, strengths, current transition, and project positioning.

# When to use
Use this skill for:
- 자기소개 작성
- 경력 요약
- 경력기술서 문장 작성
- 이력서 보정
- LinkedIn About / 헤드라인 작성
- 프로필 소개문 작성
- 커리어 전환 설명
- "현재 어떤 일을 하는 사람인가" 설명
- 포지셔닝 문장 작성

Do not use this skill for:
- 일반 코드 디버깅
- 라이브러리 설치
- 사용자 맥락이 필요 없는 단순 문장 수정
- 기술 비교만 필요한 요청

# Writing principles
## Business language first
Explain with business meaning first, not technical implementation.

## Management logic over feature listing
Always explain what the structure helps managers understand, decide, or act on.

## Grounded tone
Keep the tone practical, credible, and evidence-based.

## No overclaiming
Do not make independent projects sound like fully commercialized enterprise products unless the user explicitly asks for that framing.

## Coherent transition
The career narrative should feel like:
field sales experience → operational insight → data-based structuring → sales planning / SFE / CRM system design

# Recommended narrative flow
Unless the user asks otherwise, use this order:
1. Career foundation
2. What the user learned from field experience
3. Why the user moved toward planning / system work
4. What the user is building now
5. What value that work creates
6. Which role that connects to

# Output patterns
## Pattern A: Short profile
- current positioning
- career foundation
- current focus
- business value

## Pattern B: Resume / LinkedIn summary
- identity
- experience base
- current project direction
- future role alignment

## Pattern C: Role-fit summary
- what the user has done
- why that matters
- what the user is doing now
- why the user fits the target role

# Preferred phrasing
Use phrasing like:
- structures field insight into manageable operating logic
- turns sales performance into a manageable process, not just an outcome
- connects activity, target, and result into one management framework
- helps managers diagnose status and decide next actions
- builds common standards for coaching and performance dialogue
- translates field complexity into decision-ready reporting

# Avoid phrasing
Avoid:
- disruptive platform
- revolutionary SaaS
- full-stack developer as main identity
- generic AI transformation wording
- feature dumping without management meaning

# Special caution
When writing self-introduction or career summaries:
- do not hide the independent project period
- do not understate the family business period
- do not exaggerate technical depth
- do not make the user sound junior
- keep the user positioned as a senior field-experienced operator moving into planning and operating-system design
