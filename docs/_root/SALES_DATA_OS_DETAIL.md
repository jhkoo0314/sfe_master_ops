# Sales Data OS 상세설명서

작성일: 2026-03-17

## 1. 한 줄 요약

Sales Data OS는 제약 영업 데이터를 **표준화 → KPI 계산 → 검증(OPS) → 인텔리전스(RADAR) → 보고서(Builder)**로 이어주는 운영 시스템입니다.

---

## 2. 왜 OS인가 (의미)

이 프로젝트는 단순 대시보드가 아니라,  
**데이터가 들어오면 표준화부터 결과 전달까지 자동으로 이어지는 “운영 체계”**입니다.

즉, 한 번 세팅하면
- 데이터가 들어오고
- 표준화되고
- KPI가 계산되고
- 검증을 거쳐
- 분석/결정 지원으로 전달되고
- 최종 보고서가 만들어집니다.

이 흐름이 반복적으로 작동한다는 점에서 OS라는 이름을 사용합니다.

---

## 3. 핵심 원칙 (짧게)

1. 시스템 전체 이름은 **Sales Data OS**
2. OPS는 **Validation / Orchestration Layer** (검증/운영 통제 레이어)
3. KPI 계산은 **단일 소스** (`modules/kpi/*`)
4. Builder는 **렌더 전용 (계산 금지)**
5. 모듈 간 전달은 **Result Asset** 기준

### KPI 단일 소스의 뜻
- KPI는 여기저기서 따로 계산하지 않습니다.
- 같은 숫자를 여러 파일에서 다시 만들면 기준이 흔들리기 때문입니다.
- 그래서 공식 KPI 계산은 `modules/kpi/*`만 담당합니다.

---

## 4. 전체 흐름 (공식)

`원천데이터 → Adapter → Module/Core Engine → Result Asset → OPS → Intelligence(RADAR) → Builder`

여기서 OPS는 별도의 “최상위 시스템”이 아니라,  
모듈 결과를 다음 단계로 넘기기 전에 **검증하고 조정하는 운영 레이어**입니다.

---

## 5. 레이어별 설명 (쉽게)

### 1) Data Layer (원천 데이터)
회사별 raw 데이터가 들어오는 구간입니다.  
예: CRM 활동, 매출, 목표, 처방, 마스터 데이터

역할:
- “있는 그대로” 데이터를 보관
- 아직 계산/정리하지 않음

### 2) Adapter Layer (표준화)
회사마다 다른 형식을 공통 구조로 바꾸는 단계입니다.

역할:
- raw를 표준 스키마로 변환
- 원본 의미를 보존
- 이후 단계가 통일된 형태로 처리 가능

### 3) Core Engine Layer (모듈 계산)
각 모듈의 공식 계산 단계입니다.  
KPI 계산은 반드시 `modules/kpi/*`가 단일 소스입니다.

역할:
- KPI 계산
- 모듈 결과 자산(Result Asset) 생성
- 다음 단계가 재계산 없이 쓸 수 있도록 정리

### KPI 모듈
**Sales Data OS에서 숫자를 실제로 계산하는 기준 모듈**

의의:
- 영업 데이터를 “해석 가능한 숫자”로 바꿉니다.
- 각 모듈이 같은 기준으로 KPI를 보게 해줍니다.
- 나중에 숫자가 바뀌어도 한 곳만 수정하면 되게 합니다.

쉽게 말하면:
- 원본 데이터는 재료
- KPI 모듈은 계산기
- 결과 화면은 계산 결과를 보여주는 곳

핵심:
- 공식 KPI 계산 소스만 담당
- 같은 KPI를 여러 곳에서 중복 계산하지 않음
- 계산된 결과를 다음 모듈이 그대로 사용하게 함

하지 않는 것:
- 검증 판단
- 보고서 렌더링
- 원본 데이터 형식 변환

### 4) Validation / Orchestration Layer (OPS)
OPS는 **검증과 전달 판단**만 담당하는 운영 모듈입니다.

의미:
- 모듈이 만든 결과를 바로 다음 단계로 넘기지 않고 먼저 확인합니다.
- 데이터가 맞는지, 형식이 맞는지, 넘겨도 되는지 판단합니다.
- 전체 흐름이 끊기지 않도록 순서를 통제합니다.

역할:
- 품질 검증
- 매핑 검증
- 다음 단계로 넘길지 판단
- 실행 순서/상태 통제

하지 않는 것:
- KPI 계산
- 보고서 렌더링
- 시스템 전체 설명 역할

### 5) Intelligence Layer (RADAR 포함)
검증 승인된 결과 자산을 바탕으로  
신호 탐지와 우선순위 정리를 수행합니다.

역할:
- signal detection
- issue prioritization
- decision option 템플릿화

하지 않는 것:
- KPI 재계산
- 원인 단정
- 현장 실행 자동 결정

### 6) Presentation Layer (Builder)
최종 HTML 보고서를 만드는 단계입니다.

역할:
- 이미 계산된 payload를 읽어 HTML로 렌더링
- 결과물 다운로드/미리보기 제공

하지 않는 것:
- KPI 계산
- raw 해석

---

## 6. 모듈별 역할과 의의

### 1) CRM 모듈
**영업 활동 데이터를 행동 KPI 자산으로 만드는 출발 모듈**

의의:
- 이후 모든 모듈이 참고하는 기준선을 만듭니다.
- 단순 기록이 아니라 “행동 KPI”를 만들어 냅니다.

핵심:
- 공식 KPI 계산 소스: `modules/kpi/crm_engine.py`

### 2) OPS 모듈
**모듈 결과를 검증하고 다음 단계로 넘길지 판단하는 운영 모듈**

의의:
- Sales Data OS 안에서 흐름을 관리하는 “관문” 역할입니다.
- 계산을 새로 만들지 않고, 이미 나온 결과가 안전한지 확인합니다.
- 오류가 있으면 다음 단계로 넘기지 않고 막아줍니다.

핵심:
- 품질 검증
- 매핑 검증
- 전달 가능 여부 판단
- 실행 순서 통제

하지 않는 것:
- KPI 계산
- 분석 결과 재작성
- 보고서 렌더링

### 3) KPI 모듈
**제약 영업 숫자를 공식 기준으로 계산하는 모듈**

의의:
- Sales Data OS 전체에서 숫자의 기준을 하나로 맞춥니다.
- CRM, Sandbox, Territory, Prescription에서 쓰는 KPI의 출처가 됩니다.
- 계산 규칙이 바뀌어도 한 곳에서만 수정하면 됩니다.

쉽게 말하면:
- KPI 모듈은 “정답 계산기”입니다.
- 다른 모듈은 이 계산 결과를 가져다 쓰기만 합니다.

핵심:
- 단일 계산 소스
- 중복 계산 금지
- 모듈 간 숫자 기준 통일

하지 않는 것:
- 검증
- 화면 표시
- 재해석

관련 파일:
- `modules/kpi/crm_engine.py`
- `modules/kpi/sandbox_engine.py`
- `modules/kpi/territory_engine.py`
- `modules/kpi/prescription_engine.py`

### 4) Prescription 모듈
**도매 → 약국 → 병원 흐름을 추적해 검증 자산을 만드는 모듈**

의의:
- 처방 흐름의 정합성을 점검하고 결과 자산으로 만듭니다.
- 흐름 기반 검증 관점 강화

핵심:
- 공식 KPI 계산 소스: `modules/kpi/prescription_engine.py`

### 5) Sandbox 모듈
**검증 승인된 자산을 받아 통합 분석 자산을 만드는 모듈**

의의:
- KPI를 다시 계산하지 않고도 통합 관점 분석을 제공합니다.
- 지점/담당자 분석 구조를 안정적으로 운영합니다.

핵심:
- 공식 KPI 계산 소스: `modules/kpi/sandbox_engine.py`

### 6) Territory 모듈
**활동과 공간 정보를 결합해 권역 실행 자산을 만드는 모듈**

의의:
- “지역/담당자 실행 구조”를 분석할 수 있게 만듭니다.
- 지도 기반 실행 관점을 제공합니다.

핵심:
- 공식 KPI 계산 소스: `modules/kpi/territory_engine.py`

### 7) RADAR 모듈
**KPI 결과 기반 신호 탐지와 우선순위 정리 모듈**

의의:
- 의사결정 지원을 위한 “신호 요약”을 제공합니다.
- 분석 결과를 행동 옵션으로 정리하는 레이어입니다.

핵심:
- KPI 재계산 금지

### 8) Builder 모듈
**최종 결과물을 보여주는 렌더링 모듈**

의의:
- 보고서 “표현”만 담당하여 역할이 분리됩니다.
- 템플릿 중심으로 결과를 전달합니다.

핵심:
- 계산 금지
- payload 소비 전용

---

## 7. Result Asset의 의미

Result Asset은  
**모듈 계산 결과를 다음 단계로 안전하게 넘기는 표준 산출물**입니다.

원칙:
- 다음 단계는 Result Asset만 소비
- Result Asset 이후 단계는 계산 재실행 금지
- Builder는 Result Asset을 표현만 함

---

## 8. 포트폴리오 관점 핵심 메시지

이 프로젝트는 제약 영업기획 직무 관점에서 다음을 보여줍니다.

1. **데이터 표준화 체계 설계 능력**
2. **KPI 단일 소스 운영 원칙**
3. **검증 기반 운영 통제 구조**
4. **분석/의사결정 지원 레이어 분리**
5. **보고서 생산 체계화**

---

## 9. 금지할 오해 (중요)

- OPS가 시스템 전체다 (아님)
- OPS는 검증/운영 통제 모듈이다 (맞음)
- OPS가 KPI 계산한다 (아님)
- Builder가 계산한다 (아님)
- RADAR가 KPI 재계산한다 (아님)

---

## 10. 결론

Sales Data OS는  
**데이터 → 계산 → 검증 → 인텔리전스 → 표현**을  
한 흐름으로 묶어주는 제약 영업 운영 체계입니다.

OPS는 그 안의 **검증/운영 통제 레이어**이고,  
모든 계산은 모듈의 KPI 엔진에서만 수행됩니다.
