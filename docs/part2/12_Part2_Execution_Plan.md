# Part 2 실행계획서

작성일: 2026-03-11

## 0. 문서 목적

이 문서는 Part 1 완료 이후  
Part 2를 실제로 어떤 순서로 진행할지 정하는 실행계획서다.

핵심 목적은 3가지다.

1. 무엇을 먼저 만들지 우선순위를 고정한다.
2. 각 단계의 완료 기준을 분명히 한다.
3. 새 기능을 붙여도 기존 OPS 세계관이 흔들리지 않게 한다.

---

## 1. Part 2 한 줄 목표

`OPS를 분석 묶음이 아니라 실제 운영 허브로 키운다.`

쉽게 말하면 Part 2는

- 보고서를 더 많이 만드는 단계

가 아니라

- 매일 안정적으로 돌고
- 실패를 빨리 알 수 있고
- 현업이 바로 움직일 수 있는 구조

로 가는 단계다.

---

## 2. Part 2에서 지킬 원칙

Part 2에서도 아래 원칙은 그대로 유지한다.

```text
원천데이터 -> Adapter -> Module -> Result Asset -> OPS -> Builder
```

즉:

- raw를 새 화면이 직접 읽지 않는다
- OPS가 직접 분석 계산 엔진처럼 비대해지지 않는다
- 새 모듈도 Result Asset 또는 검증 결과를 재료로 쓴다
- Builder는 끝까지 표현 단계로 유지한다

---

## 3. 이번 단계에서 하지 않을 것

Part 2 범위를 지키기 위해 아래는 우선순위에서 뺀다.

1. 보고서 디자인 전면 개편
2. 모든 회사를 위한 완전 범용 설정 시스템 만들기
3. Builder에 새 계산 로직 계속 추가하기
4. 기존 모듈 로직을 대규모로 다시 뜯어고치기

즉 Part 2는  
`운영화와 관제 강화`가 중심이다.

---

## 4. 최우선 작업 묶음

## 4.1 Workstream A. Master Quality / Intake Gate

### 목적

실행 전에 가장 먼저

- 원천 파일이 들어왔는지
- 필수 컬럼이 맞는지
- 병원/담당자/품목 매핑이 어느 정도 안전한지

를 확인하는 입구 점검 모듈을 만든다.

### 왜 먼저 하는가

실제 운영에서는 분석 실패보다  
`입력 데이터 이상`이 더 먼저 문제를 만든다.

지금 구조는 중간 검증은 강해졌지만,  
입구 상태를 한 장으로 보는 기능은 아직 약하다.

### 권장 산출물

- `intake_result_asset.json`
- `intake_ops_evaluation.json`
- `intake_validation_summary.json`
- `intake_gate_preview.html`

### 권장 위치

- `modules/intake/`
- `result_assets/intake_result_asset.py`
- `ops_core/api/intake_router.py`
- `scripts/validate_intake_with_ops.py`

### 완료 기준

1. 회사 코드 기준으로 intake 점검을 단독 실행할 수 있다.
2. 필수 파일 누락, 컬럼 누락, 매핑 위험을 점수 또는 등급으로 보여준다.
3. OPS가 intake 결과를 읽고 다음 실행 가능 여부를 판단할 수 있다.
4. Builder 미리보기로 사람이 바로 확인할 수 있다.

---

## 4.2 Workstream B. Ops Monitoring / Daily Watch

### 목적

운영자가 매일 한 화면에서

- 어느 회사가 성공했는지
- 어떤 단계가 실패했는지
- 마지막 실행 시각이 언제인지
- 어떤 보고서가 최신인지

를 보게 한다.

### 왜 필요한가

지금은 실행은 가능하지만,  
운영 관제는 수동 확인 비중이 남아 있다.

운영 직전 다음 단계는  
`잘 돌아간다`보다 `지금 상태를 한눈에 안다`가 더 중요하다.

### 권장 산출물

- `daily_watch_snapshot.json`
- `daily_watch_preview.html`
- 콘솔 안 `운영 관제` 탭 또는 별도 화면

### 권장 위치

- `ops_core/monitoring/`
- `scripts/build_daily_watch.py`
- `ui/console_tabs.py` 또는 새 전용 화면

### 완료 기준

1. 회사별 마지막 실행 성공/실패를 모아 볼 수 있다.
2. 단계별 상태를 배지로 보여준다.
3. 마지막 생성 보고서 시각을 보여준다.
4. 최근 실패 건을 바로 열어 원인을 따라갈 수 있다.

---

## 4.3 Workstream C. 배치 자동화 / 실패 알림

### 목적

사람이 매번 수동으로 누르지 않아도

- 정해진 시각 실행
- 실패 시 재시도
- 실패 시 알림

이 가능하게 한다.

### 왜 필요한가

Part 1은 `실행 가능한 구조`를 만든 단계였다.
Part 2는 `운영 가능한 구조`로 가야 한다.

운영 가능의 핵심은

- 자동 실행
- 실패 감지
- 빠른 알림

이다.

### 권장 범위

- Windows Task Scheduler 또는 배치 진입점
- 회사 코드별 실행 배치
- 실패 로그 저장
- 간단한 알림 채널
  - 메일
  - 메신저
  - 로그 파일 우선도 가능

### 완료 기준

1. 최소 1개 회사에 대해 예약 실행이 돈다.
2. 실패하면 로그와 실패 이유가 남는다.
3. 재시도 규칙이 문서화된다.
4. 운영자가 실패 사실을 수동 확인 없이 알 수 있다.

---

## 4.4 Workstream D. Action Queue / Field Recommendation

### 목적

여러 모듈 결과를 합쳐서  
현업이 바로 움직일 수 있는 우선순위 목록을 만든다.

쉽게 말하면:

- 누구를 먼저 코칭할지
- 어떤 병원을 먼저 다시 볼지
- 어떤 권역을 먼저 손볼지

를 한 장의 행동 큐로 만든다.

### 왜 마지막인가

이 기능은 가치가 크지만,
입구 점검과 운영 관제가 먼저 안정돼야 신뢰할 수 있다.

즉 Action Queue는  
운영 허브 위에 올라가는 `행동 우선순위 레이어`다.

### 권장 산출물

- `action_queue_result_asset.json`
- `action_queue_preview.html`
- 회사별 액션 우선순위 표

### 재료 원칙

반드시 아래를 재료로 삼는다.

- CRM Result Asset
- Sandbox 결과
- Prescription 결과
- Territory 결과
- OPS 평가 결과

raw를 직접 다시 읽지 않는다.

### 완료 기준

1. 우선순위 기준이 문서화된다.
2. 우선순위 목록이 회사 코드 기준으로 생성된다.
3. 왜 이 항목이 상위인지 설명 문장이 함께 나온다.
4. 현업이 바로 보는 HTML 또는 표가 나온다.

---

## 4.5 Workstream E. 회사별 규칙 / 임계치 외부화

### 목적

현재 코드 안에 있는 일부 기준값을  
회사별 설정 파일이나 정책 테이블로 조금씩 빼낸다.

### 왜 필요한가

새 회사를 붙일 때  
코드 수정 없이 조정 가능한 범위를 늘려야  
공통 틀의 가치가 더 커진다.

### 우선 외부화 후보

- 품질 배지 기준
- 경고 임계치
- 실행 허용 기준
- 보고서 표시용 회사 메타 정보

### 완료 기준

1. 최소 1개 회사 정책 파일을 별도로 둔다.
2. 코드 수정 없이 일부 임계치를 바꿀 수 있다.
3. 문서에 `어디까지 설정으로 바꾸고 어디는 코드로 남기는지` 기준이 적힌다.

---

## 5. 권장 실행 순서

Part 2는 아래 순서가 가장 안전하다.

1. Workstream A. Intake Gate
2. Workstream B. Daily Watch
3. Workstream C. 배치 자동화 / 실패 알림
4. Workstream D. Action Queue
5. Workstream E. 규칙 / 임계치 외부화

이 순서가 좋은 이유:

- 먼저 입구를 안정화하고
- 그 다음 운영 상태를 보이게 하고
- 그 다음 자동 실행을 붙이고
- 마지막에 현업 액션 가치와 설정 확장을 올리는 흐름이기 때문이다.

---

## 6. 8주 권장 일정안

## 6.1 1단계: 1~2주차

목표:

- Intake Gate 최소 버전 완성

할 일:

- intake 표준 점검 항목 정의
- 결과 자산 정의
- 검증 스크립트 추가
- 간단한 HTML 미리보기 추가

완료 판단:

- 회사 1곳 기준으로 단독 실행 가능
- 누락/매핑 위험을 등급으로 보여줌

## 6.2 2단계: 3~4주차

목표:

- Daily Watch 최소 버전 완성

할 일:

- 회사별 마지막 실행 상태 수집
- 실패/성공 요약 JSON 만들기
- 콘솔 또는 전용 HTML에서 상태판 만들기

완료 판단:

- 운영자가 한 화면에서 회사별 상태 확인 가능

## 6.3 3단계: 5~6주차

목표:

- 예약 실행과 실패 알림 연결

할 일:

- 예약 실행 스크립트 정리
- 실패 로그 저장
- 알림 채널 1개 연결

완료 판단:

- 최소 1개 회사 자동 실행
- 실패 시 운영자가 바로 인지

## 6.4 4단계: 7~8주차

목표:

- Action Queue 1차 버전
- 일부 정책 외부화 시작

할 일:

- 우선순위 계산 기준 정의
- 회사별 액션 목록 생성
- 임계치 일부를 설정 파일로 이동

완료 판단:

- 현업이 바로 볼 수 있는 액션 목록 생성
- 회사별 기준 일부를 코드 밖에서 조정 가능

---

## 7. 각 단계 공통 완료 기준

Part 2의 모든 작업은 아래 6가지를 같이 만족해야 완료로 본다.

1. 회사 코드 기준 경로를 사용한다.
2. Result Asset 또는 검증 결과를 중심으로 연결한다.
3. Builder는 표현 단계로만 유지한다.
4. 회귀 테스트 또는 최소 검증 스크립트가 있다.
5. 문서가 같이 갱신된다.
6. 운영 콘솔 또는 실행 문서에서 실제 사용 경로가 설명된다.

---

## 8. 권장 성공지표

Part 2가 잘되고 있는지 보려면 아래를 추적하는 것이 좋다.

### 운영 지표

- 회사별 일일 실행 성공률
- 실패 후 재시도 성공률
- 마지막 성공 실행 후 경과 시간

### 데이터 지표

- 필수 파일 누락 건수
- 매핑률 하락 감지 건수
- 전일 대비 데이터 급감 건수

### 활용 지표

- 생성 보고서 최신화 비율
- Action Queue 생성 비율
- 운영자가 관제 화면에서 바로 확인 가능한 회사 수

## 9. 주요 리스크와 대응

### 리스크 1. Part 2가 새 분석 기능 모음으로 흐를 수 있음

대응:

- Intake, Monitoring, Automation을 먼저 끝내고 나서 새 분석을 붙인다.

### 리스크 2. Builder가 다시 무거워질 수 있음

대응:

- 새 화면도 기본은 summary-first, lazy loading 원칙을 따른다.

### 리스크 3. 회사별 예외 대응이 코드 분기로 다시 늘어날 수 있음

대응:

- adapter/profile/config로 먼저 흡수하고,
- 본 모듈 로직에는 회사별 분기를 최소화한다.

### 리스크 4. 문서 없이 운영 규칙만 구두로 남을 수 있음

대응:

- 실행 규칙, 실패 기준, 알림 규칙을 문서에 같이 남긴다.

---

## 10. 최종 권고

Part 2의 핵심은  
`OPS를 실제 매일 돌리는 운영 허브로 키우는 것`이다.

따라서 가장 좋은 시작은 아래다.

1. Intake Gate 먼저 만든다.
2. Daily Watch로 운영 상태를 보이게 만든다.
3. 배치와 알림으로 수동 의존을 줄인다.
4. 그 다음에 Action Queue로 현업 가치까지 확장한다.

이 순서를 지키면  
Part 1에서 만든 안정성을 해치지 않고  
Part 2로 자연스럽게 넘어갈 수 있다.
