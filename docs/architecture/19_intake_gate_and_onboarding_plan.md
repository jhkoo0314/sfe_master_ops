# 19. Intake Gate And Onboarding Plan

## 문서 목적

이 문서는 실제 회사 raw 데이터가 들어왔을 때, Sales Data OS가 **어떻게 intake gate와 onboarding을 수행해야 하는지**를 설명하는 운영 설계 문서다.

이 문서의 목적은 세 가지다.

1. 새 회사 raw 데이터를 받는 입구를 표준화한다.
2. 사용자가 회사 등록과 raw 업로드만 하면, 그 뒤는 시스템이 최대한 자동 처리하도록 설계한다.
3. Adapter를 회사마다 계속 수정하지 않고, 앞단에서 raw를 정리해 고정된 Adapter로 넘기는 구조를 정의한다.

중요:

- 이 문서는 테스트용 raw 생성기 문서가 아니다.
- 실제 운영에서 가장 중요한 앞단 구간을 설명한다.
- Adapter 이후 파이프라인이 아니라, `raw -> adapter 이전 구간`을 다룬다.

---

## 1. 왜 이 문서가 중요한가

현재 Sales Data OS는 Adapter 이후 파이프라인이 상당 부분 준비돼 있다.

- Adapter
- Core Engine / KPI Engine
- Validation Layer (OPS)
- Intelligence
- Builder

즉 raw가 Adapter가 읽을 수 있는 수준까지 정리되면, 그 뒤는 이미 정해진 코드 흐름으로 계산과 결과 생성이 가능하다.

문제는 앞단이다.

실제 회사 raw는 아래처럼 지저분할 수 있다.

- 컬럼명이 제각각
- 날짜 형식이 다름
- 월 형식이 다름
- 병원코드/담당자코드 일부 누락
- 제품코드 없이 제품명만 존재
- 월별 파일과 일괄 파일이 섞여 있음
- 중복 행과 비정상 값이 섞여 있음

이 상태에서 회사마다 Adapter를 계속 고치는 방식은 범용 운영 시스템으로 유지할 수 없다.

따라서 핵심은 이 구조다.

`company raw -> intake gate -> onboarding -> adapter -> existing pipeline`

운영 관점 핵심:

- 바깥에 보이는 intake/onboarding 엔진은 하나여야 한다.
- 회사마다 새 스크립트를 만드는 구조는 지양해야 한다.
- 새 회사 차이는 `시나리오 + 매핑 + 룰` 업데이트로 흡수해야 한다.

---

## 2. 핵심 개념 정의

### 2.1 Intake Gate

Intake Gate는 **새 raw 데이터를 처음 받는 입구 심사 + 자동 보정 레이어**다.

하는 일:

- 파일 종류 확인
- 필수 입력 여부 확인
- 컬럼 구조 확인
- 값 형식 확인
- 자동 수정 가능한 부분 보정
- 수정 제안 생성
- Adapter 전달 가능 여부 판단

하지 않는 일:

- KPI 계산
- 최종 검증 결과 판정
- Builder 렌더링

즉 Intake Gate는 단순 검사기가 아니라, raw를 Onboarding 가능한 상태로 만드는 첫 레이어다.

구현 위치 원칙:

- 책임상 `Intake Gate`와 `Onboarding`은 `modules/intake` 아래에 두는 것이 맞다.
- 현재 `ops_core`는 실행 제어 경로로 유지하더라도, intake 자체의 책임은 `ops_core`가 아니라 `modules` 쪽으로 본다.

### 2.2 Onboarding

Onboarding은 Intake Gate 통과 후, **그 회사 raw를 실제 시스템에 연결하는 단계**다.

하는 일:

- 회사 기준 source 경로 확정
- 컬럼 매핑 확정
- 월별/일괄 구조 정리
- Adapter 입력용 중간 정리본 생성
- 회사별 설정 저장
- 첫 실행 검증 연결

즉 Onboarding은 “이제 이 회사 raw를 Sales Data OS에서 안정적으로 돌릴 수 있다” 상태를 만드는 단계다.

구현 위치 원칙:

- `Onboarding`도 별도 실행 제어가 아니라 실제 raw 처리 책임이므로 `modules/intake` 아래에서 함께 다루는 것이 맞다.
- `ops_core/workflow`는 intake를 호출하는 orchestration 레이어로만 남겨야 한다.

### 2.3 관계 정리

둘은 분리된 책임이지만 실제 운영에서는 한 흐름으로 이어져야 한다.

`업로드 -> intake gate -> onboarding -> adapter`

---

## 3. 사용자 워크플로우 기준 정리

사용자 기준으로는 가능한 단순해야 한다.

### 사용자가 직접 하는 것

1. 회사 등록
2. raw 업로드

### 시스템이 자동으로 하는 것

1. Intake Gate
2. Onboarding
3. Adapter 실행
4. 이후 기존 파이프라인 실행

즉 사용자가 raw를 여러 번 손으로 고쳐 다시 올리는 구조가 아니라, 시스템이 먼저 최대한 보정하고 제안하고 넘기는 구조가 맞다.

---

## 4. Intake Gate가 해야 하는 일

### 4.1 파일 심사

확인할 것:

- 필수 파일이 있는지
- 파일 형식이 맞는지 (`xlsx`, `csv`)
- 월별 파일인지 일괄 파일인지
- 중복 업로드가 있는지

예시:

- `sales_202501.xlsx` -> 월별 파일로 인식
- `sales_raw.xlsx` -> 일괄 파일로 인식

### 4.2 컬럼 심사

확인할 것:

- 병원 식별 컬럼 존재 여부
- 담당자 식별 컬럼 존재 여부
- 제품 식별 컬럼 존재 여부
- 날짜 또는 월 컬럼 존재 여부
- 금액/수량 컬럼 존재 여부

중요:

컬럼 이름이 표준 이름과 다르더라도, 의미를 추정할 수 있으면 후보 매핑을 만들 수 있어야 한다.

### 4.3 값 심사

확인할 것:

- 월 형식이 정상인지
- 날짜가 파싱 가능한지
- 코드값 공백 비율이 과도한지
- 중복 행 비율이 높은지
- 금액/수량이 비정상 값인지

### 4.4 자동 수정

자동 수정 가능한 것은 Intake Gate에서 바로 고친다.

예:

- `2025-01` -> `202501`
- 공백 제거
- 컬럼명 trim
- 중복 헤더 정리
- csv/xlsx 형식 통일
- 월별 파일 병합
- `Y/N`, `True/False`, `1/0` 같은 표현 통일

### 4.5 수정 제안

자동 수정이 어려운 것은 제안을 만든다.

예:

- “이 컬럼을 병원명 컬럼으로 추천합니다.”
- “담당자 ID가 없어 담당자명 기준 임시 연결을 추천합니다.”
- “제품코드가 없어 제품명 기준 매핑으로 진행해야 합니다.”
- “누락률이 높아 결과 신뢰도가 낮을 수 있습니다.”

### 4.6 Intake Gate 판정

마지막에는 아래 중 하나로 판정한다.

- `ready`
  - 바로 onboarding 가능
- `ready_with_fixes`
  - 자동 수정 후 onboarding 가능
- `needs_review`
  - 제안은 가능하지만 사람 확인 필요
- `blocked`
  - 핵심 입력 부족으로 onboarding 불가

---

## 5. Onboarding이 해야 하는 일

### 5.1 회사 연결 확정

- `company_key` 고정
- source 경로 확정
- 월별/일괄 입력 구조 확정

### 5.2 매핑 확정

- 어떤 raw 컬럼이 어떤 표준 의미인지 확정
- 회사별 매핑 저장
- 다음 업로드 때 재사용 가능하도록 유지

### 5.3 Adapter 전달용 정리본 생성

Onboarding은 raw 원본을 직접 고치기보다, Adapter가 읽을 수 있는 **중간 정리본**을 만드는 것이 더 안전하다.

예:

- `staged_source`
- `onboarding_ready_source`

즉 원본은 보존하고, 정리본을 만들어 Adapter로 넘긴다.

### 5.4 첫 실행 검증

Onboarding이 끝나면 최소 실행모드로 바로 검증한다.

권장:

1. `crm_to_pdf` 또는 `crm_to_sandbox`
2. 필요 시 `integrated_full`

이 단계에서 처음으로 “이 회사 데이터가 실제로 돌아간다”를 확인한다.

---

## 6. Adapter를 왜 회사마다 수정하면 안 되는가

단일 회사 전용 시스템이라면 회사마다 Adapter를 맞춰도 된다.

하지만 Sales Data OS는 여러 회사를 수용해야 한다.

따라서 아래 원칙이 필요하다.

1. Adapter는 최대한 고정
2. 회사별 차이는 Intake/Onboarding에서 흡수
3. 회사별 차이는 설정과 매핑으로 관리
4. Adapter는 “정리된 입력”을 받는 번역기로 유지

즉 범용 시스템의 목표는 아래다.

**회사 raw는 제각각이어도, Intake/Onboarding이 정리해서 결국 Adapter에는 항상 비슷한 형태로 넣는다.**

---

## 7. 자동 수정 / 제안 / 사람 확인의 경계

### 7.1 자동 수정 가능

- 월 형식 통일
- 날짜 형식 통일
- 컬럼명 공백 제거
- 기본 중복 제거
- 월별 병합
- 파일 형식 변환

### 7.2 제안 가능

- 병원명 컬럼 후보
- 담당자 컬럼 후보
- 제품명/제품코드 후보
- 병원명 기준 역매핑 제안
- 임시 매핑 전략 제안

### 7.3 사람 확인 필요

- 핵심 식별 컬럼이 완전히 없음
- 병원/담당자/제품 의미를 추정할 수 없음
- 누락률이 너무 높아 결과 신뢰성이 무너짐
- 서로 다른 해석 후보가 여러 개라 자동 결정이 위험함

---

## 8. 운영 결과로 사용자에게 보여줘야 하는 것

사용자는 개발자가 아니므로, Intake/Onboarding 결과를 쉽게 봐야 한다.

### 보여줘야 하는 것

1. 어떤 파일을 인식했는지
2. 어떤 컬럼을 어떤 의미로 읽었는지
3. 어떤 부분을 자동 수정했는지
4. 어떤 부분은 제안만 했는지
5. 지금 onboarding 가능한지
6. Adapter로 넘긴 최종 상태가 어떤지

### 보여주면 안 되는 것

- 내부 구현용 코드 이름 남발
- 스택트레이스 위주 오류 메시지
- “실패”만 던지고 끝내는 메시지

---

## 9. 성공 기준

이 설계가 성공하려면 아래가 가능해야 한다.

1. 사용자는 회사 등록과 raw 업로드만 한다.
2. 시스템은 Intake Gate에서 가능한 부분을 자동 수정한다.
3. 시스템은 수정 불가 부분을 사람이 이해할 수 있는 문장으로 제안한다.
4. Onboarding은 회사별 매핑과 연결을 저장한다.
5. Adapter는 회사마다 수정하지 않고 재사용된다.
6. Adapter 이후 기존 파이프라인은 그대로 활용된다.

---

## 10. 최종 정리

Sales Data OS에서 지금 가장 중요한 새 구조는 아래다.

`회사 등록 -> raw 업로드 -> intake gate -> onboarding -> adapter -> 기존 파이프라인`

핵심은 다음 두 줄이다.

- Intake Gate는 raw를 막는 문이 아니라, onboarding 가능한 상태로 정리하는 입구 레이어다.
- Onboarding은 그 정리된 raw를 실제 운영 파이프라인에 붙이는 연결 레이어다.

한 줄 요약:

**새 회사 온보딩의 핵심은 Adapter를 회사마다 고치는 것이 아니라, Intake Gate와 Onboarding이 회사 raw를 정리해서 고정된 Adapter로 넘기게 만드는 것이다.**
