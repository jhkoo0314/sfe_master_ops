# SFE OPS Master PRD

작성일: 2026-03-10

## 0. 이 문서의 역할

이 문서는 SFE OPS를 아무것도 없는 상태에서 다시 시작하기 위한 `단일 PRD`다.

앞으로 SFE OPS를 설명할 때 가장 먼저 보는 문서는 이 문서다.

쉽게 말하면:

- 왜 이 프로젝트를 하는지
- 정확히 무엇을 만들려는지
- 무엇은 일부러 안 하는지
- 5개 모듈을 어떤 세계관으로 묶는지

를 한 번에 고정하는 문서다.

---

## 1. 제품 한 줄 정의

`SFE OPS는 5개 SFE 모듈을 하나로 합치는 시스템이 아니라, 각 모듈이 만든 Result Asset을 공통 규칙으로 검증하고 연결하는 중앙 운영 엔진이다.`

---

## 2. 왜 필요한가

SFE 문제는 개별 기능이 없어서 생기는 문제가 아니다.

진짜 문제는 아래다.

1. 모듈마다 입력 구조와 키 체계가 다르다.
2. 원천데이터 형식이 회사마다 다르다.
3. 한 모듈의 결과가 다른 모듈에서 바로 재사용되기 어렵다.
4. raw 중심으로 생각하면 중복 해석이 반복된다.
5. 특정 회사 구조에 빨리 종속되면 범용 설계가 무너진다.

즉 SFE OPS의 목적은 새 기능을 하나 더 만드는 것이 아니라,
서로 흩어진 모듈을 `실무형 운영 구조`로 연결하는 것이다.

---

## 3. 해결하려는 문제

SFE OPS는 아래 문제를 해결한다.

1. 공통 상태 판단 부재
2. 모듈 간 연결 기준 부재
3. 키 정합성 관리 부재
4. 품질 게이트 부재
5. raw 중심 사고로 인한 중복 해석
6. 특정 회사 기준에 빨리 종속되는 설계

---

## 4. SFE OPS 세계관

이 프로젝트의 세계관은 아래 문장으로 고정한다.

1. 중심은 `Sandbox`가 아니라 `OPS Engine`이다.
2. 5개 모듈은 `OPS를 통해 연결되는 방사형 구조`다.
3. 모듈은 raw를 바로 OPS에 넘기지 않는다.
4. 먼저 `adapter`가 raw를 공통 구조로 번역한다.
5. 모듈은 그 입력으로 `Result Asset`을 만든다.
6. OPS는 `Result Asset`만 검증하고 다음 연결을 판단한다.
7. Sandbox는 핵심 분석엔진이지만 전체 허브는 아니다.

쉽게 말하면:

- `adapter` = 회사별 차이를 흡수하는 입구
- `module` = 자기 결과를 만드는 처리기
- `OPS` = 결과를 보고 연결을 판단하는 중앙 엔진

---

## 5. 무엇을 만드는가

SFE OPS는 아래 5개를 만드는 프로젝트다.

1. 모듈 공통 연결 규칙
2. 공통 키 거버넌스
3. Result Asset 중심 운영 구조
4. 품질/상태/연결 판단 체계
5. 5개 모듈의 재사용 가능한 연결 구조

즉 이 프로젝트는 `기능 모음`이 아니라 `운영 구조`를 만든다.

---

## 6. 무엇을 만들지 않는가

아래는 일부러 하지 않는다.

1. 5개 모듈을 하나의 거대한 단일 시스템으로 합치지 않는다.
2. Sandbox를 전체 허브로 고정하지 않는다.
3. raw를 OPS가 직접 해석하게 만들지 않는다.
4. 특정 회사 맞춤 키를 처음부터 정답처럼 고정하지 않는다.
5. adapter 없이 pipeline/service부터 키우지 않는다.
6. 실데이터가 오기 전부터 회사 맞춤 규칙을 과도하게 만들지 않는다.

---

## 7. 5개 모듈의 공식 역할

### 7.1 Behavior CRM

- 역할: 행동 기반 자산 생산
- 핵심 가치: 선행행동, KPI, 행동프로파일
- OPS 안에서의 위치: 출발 자산 생산 모듈

### 7.2 SFE Sandbox

- 역할: 통합 분석 자산 생산
- 핵심 가치: CRM, 실적, 목표, 필요 시 Prescription까지 통합 분석
- OPS 안에서의 위치: 핵심 분석엔진

### 7.3 Territory Optimizer

- 역할: 공간 실행 자산 생산
- 핵심 가치: 동선, 커버리지, 배치 효율
- OPS 안에서의 위치: 후속 Allocation 모듈

### 7.4 Prescription Data Flow

- 역할: 흐름 검증 자산 생산
- 핵심 가치: 도매 -> 약국 -> 병원 흐름 추적, 실적 검증
- OPS 안에서의 위치: Validation 모듈

### 7.5 HTML Builder

- 역할: 사람이 읽는 결과물 생산
- 핵심 가치: 보고서/웹슬라이드 생성
- OPS 안에서의 위치: 표현과 전달 모듈

---

## 8. 공식 구현 원칙

모든 구현은 아래 순서를 지킨다.

1. 세계관 확인
2. 공통 키 원칙 확인
3. raw adapter 설계
4. adapter 구현
5. module 내부 서비스 구현
6. Result Asset 생성
7. OPS 평가 연결
8. 다음 모듈 handoff 연결

특히 중요한 문장은 아래다.

`원천데이터 -> adapter -> module -> Result Asset -> OPS`

이 순서를 지키는 것이 SFE OPS의 핵심이다.

---

## 9. 공식 통합 방향

SFE OPS의 공식 통합 순서는 아래다.

1. CRM 기반 공통 병원/지점/담당자 축 안정화
2. Prescription 범용 규칙 기반 구축
3. Sandbox 통합 분석 구조 정립
4. Territory 공간 실행 구조 정립
5. HTML Builder 범용 표현 구조 정립

즉 공식 순서는 아래다.

`CRM -> Prescription -> Sandbox -> Territory -> HTML Builder`

---

## 10. 실데이터가 없을 때의 성공 기준

지금 단계에서 성공은 `실무 정답 구현`이 아니다.

지금 성공은 아래다.

1. 공통 키 원칙이 흔들리지 않는가
2. adapter-first 순서가 지켜지는가
3. OPS가 raw가 아니라 Result Asset만 보게 되어 있는가
4. 모듈 역할이 Sandbox 중심으로 왜곡되지 않는가
5. 실데이터가 와도 큰 방향이 안 무너지게 설계되어 있는가

---

## 11. 최종 성공 기준

SFE OPS는 아래가 가능해질 때 성공으로 본다.

1. 5개 모듈이 같은 세계관 아래 설명된다.
2. 모듈 간 연결이 raw가 아니라 Result Asset 기준으로 관리된다.
3. OPS가 상태, 품질, 정합성, 연결을 공통 언어로 판단한다.
4. 회사가 달라도 같은 방식으로 adapter와 키 규칙을 적용할 수 있다.
5. 문서와 코드가 같은 기준점을 바라본다.

---

## 12. 한 줄 결론

`SFE OPS는 5개 거대한 모듈을 억지로 합치는 프로젝트가 아니라, adapter-first와 Result Asset 중심 원칙 아래 OPS를 중앙 엔진으로 세워 각 모듈을 실무형 운영 구조로 연결하는 프로젝트다.`
