# Baseline And Improvements

작성일: 2026-03-12

## 1. 문서 목적

이 문서는 HTML Builder를 `독립 서비스`로 보던 기존 시각과,
현재 역할에 맞는 `출력 자동화 유틸` 시각을 비교해 정리한다.

## 2. 현재 재평가 결론

결론은 간단하다.

`HTML Builder는 독립 제품 가치보다 출력 자동화 가치가 더 크다.`

즉:

- 핵심 계산 모듈은 아니다
- 결과 전달 효율을 높이는 보조 계층이다
- 크게 키우면 역할 대비 비용이 커진다

## 3. As-Is vs To-Be

| 구분 | 기존 과대해석 | 현재 맞는 해석 |
|---|---|---|
| 정체성 | 독립 HTML 생성 서비스 | 공통 출력 자동화 유틸 |
| 입력 | 문서 업로드/API 요청 | Result Asset 또는 승인된 payload |
| 중심 가치 | 서비스 운영성 | 전달 시간 단축 |
| 핵심 구조 | API/보안/잡/관측성 | Template Contract/Injection |
| 결과 책임 | Builder도 결과를 만듦 | 공식 결과는 모듈이 만들고 Builder는 보여줌 |
| 확장 방향 | 플랫폼화 | 얇은 템플릿 계층 유지 |

## 4. 잘한 점

- 대시보드/지도/검증 화면을 빠르게 공유할 수 있다.
- 사람이 다시 꾸미는 시간을 줄인다.
- 템플릿이 정해져 있으면 출력 품질이 덜 흔들린다.

## 5. 아쉬운 점

- 문서가 독립 API 서비스처럼 과하게 설계돼 있었다.
- 역할에 비해 보안/큐/관측성/마이그레이션 문서가 과했다.
- Builder를 계산 모듈처럼 읽을 여지가 있었다.

## 6. 개선 기준

1. 독립 서비스 문서를 걷어낸다.
2. Template Contract 문서를 중심에 둔다.
3. Builder는 `템플릿 주도형 출력 단계`라고 명시한다.
4. 역할 대비 과한 운영 설계는 범위 밖으로 둔다.

## 7. 한 줄 결론

`HTML Builder는 크게 만들수록 좋은 모듈이 아니라, 얇고 빠를수록 가치가 커지는 출력 유틸이다.`
