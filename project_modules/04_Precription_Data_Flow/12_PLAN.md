# Prescription Data Flow 독립빌드 전환 계획

## 1) 왜 다시 정리하는가
기존 문서는 Prescription을 잘 설명하고 있었지만, 읽다 보면 `추적/정산 파이프라인`보다 `운영 화면 시스템`처럼 보이는 부분이 있었다.

독립빌드 기준에서는 우선순위를 다시 고정해야 한다.

- 1순위: 독립 실행
- 2순위: 추적/검증/정산 정확성
- 3순위: 승인 가능한 결과 패키지
- 4순위: 로컬 검토 화면

## 2) 새 기준 한 줄
`Prescription은 승인된 입력을 받아 추적/정산/KPI를 계산하고, 승인된 결과 패키지를 export 하는 독립 실행형 파이프라인이다.`

## 3) 목표 상태
### 3.1 필수
- CLI 또는 배치 기준으로 한 번에 실행 가능
- tracking, share, KPI, validation, trace 산출물 생성
- `prescription_result_asset.json` 생성
- `prescription_builder_payload.json` 생성
- `prescription_ops_handoff.json` 생성

### 3.2 선택
- 로컬 검토 화면
- 업로드 보조 기능
- 수동 편집 화면

## 4) 작업 순서
### 4.1 1단계 - 입력/계약 고정
- 원천 컬럼 계약 고정
- 참조 마스터와 룰 버전 구조 고정
- 실행마다 입력 manifest 저장

### 4.2 2단계 - 핵심 엔진 안정화
- `ingest_merge`
- `mastering`
- `tracking_validation`
- `share_settlement`
- `kpi_publish`
- `validation_trace`

### 4.3 3단계 - 승인 패키지 표준화
- 상세 산출물과 공식 패키지 분리
- 승인 메타 구조 고정
- 품질 상태와 handoff 준비 상태 고정

### 4.4 4단계 - OPS 연결 준비
- OPS는 결과만 읽도록 handoff 규격 문서화
- Builder는 payload만 읽도록 payload 구조 고정

### 4.5 5단계 - 선택 기능
- 로컬 검토 화면
- 업로드 자동화
- overlap ON 고도화

## 5) 냉정한 평가
잘한 점:
- 실제 운영 질문이 분명하다.
- 검증/정산/감사 로그 사고가 강하다.
- 다른 모듈보다 실무 적용성이 높다.

아쉬운 점:
- 화면과 자동화가 제품 본체처럼 보이는 표현이 많다.
- 승인 전 상세 결과와 최종 공식 결과가 문서상 분리되어 있지 않다.
- 독립 실행 후 외부 시스템으로 넘기는 경계 문서가 없었다.

개선 기준:
- `상세 산출물`과 `공식 패키지`를 나눈다.
- `핵심 엔진`과 `검토 화면`을 나눈다.
- `독립 실행`과 `OPS handoff`를 나눈다.

## 6) 이번 문서 업데이트 범위
- `PRD`, `Runbook`, `README`, `TODO`를 독립빌드 기준으로 재작성
- `KPI Output Spec`, `Phase Gate`를 승인 패키지 기준으로 수정
- `OPS handoff`, `Boundary Policy` 문서 추가
- 데이터 사전과 QA 문서에 공식 패키지 항목 반영

## 7) 완료 판정
문서 업데이트 완료로 보려면 아래가 맞아야 한다.

1. 누구나 읽었을 때 Prescription을 독립 실행형 파이프라인으로 이해한다.
2. 로컬 화면이 선택 기능이라는 점이 분명하다.
3. 승인 패키지 3종이 정의되어 있다.
4. OPS는 승인된 결과만 읽는다는 점이 분명하다.
5. 기존 비즈니스 규칙과 새 실행 모델이 충돌하지 않는다.
