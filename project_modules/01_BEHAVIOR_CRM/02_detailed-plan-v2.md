# CRM 상세 기획안 v2 (행동 유도형/선행지표 중심)

## 1. 문서 개요

- 목적: CRM을 "의무 입력 도구"에서 "좋은 행동을 유도하는 실행 시스템"으로 재설계
- 범위: 영업 운영모델, KPI 체계, 입력 신뢰 정책, 코칭 거버넌스, 90일 도입안
- 기준일: 2026-03-01

## 2. 문제 정의 (기획 출발점)

기존 CRM의 본질적 실패는 "입력 강제"에서 발생한다.

1. 입력이 성과를 위한 수단이 아니라 보고 의무로 인식됨
2. 현장은 실적 맞추기/잘 보이기 입력으로 데이터 무결성이 무너짐
3. 관리자는 많은 숫자를 보지만 "왜 성과가 났는지"를 설명하지 못함
4. 결과지표(후행) 중심 운영으로 코칭 타이밍을 놓침

## 3. 기획 의도와 비전

### 3.1 기획 의도

"입력을 강제하지 않아도 영업사원이 고효과 행동을 하게 만드는 CRM"

### 3.2 제품 비전

"선행행동을 관리해 성과를 만들고, 후행성과로 인과를 검증하는 제약영업 운영체계"

### 3.3 핵심 원칙

1. 입력량 평가 금지, 행동 품질 평가 우선
2. 선행지표 우선 운영, 후행지표는 검증 용도
3. 모든 지표는 코칭 액션으로 환원 가능해야 함
4. 개인평가(인사)와 운영개선(코칭) 용도를 분리

## 4. KPI 아키텍처 (최종)

본 계획은 단일 체계가 아니라 3계층으로 운영한다.

### 4.1 선행행동 계층 (코어 KPI)

- HIR (High-Impact Rate): 고효과 행동 실행 품질
- RTR (Relationship Temperature Rate): 계정 관계 온도
- BCR (Behavior Consistency Rate): 실행 일관성/루틴
- PHR (Proactive Health Rate): 유효 Next Action 비율

### 4.2 운영관리 계층 (보조 KPI)

- NAR (Next Action Reliability): 기한 내 Next Action 이행 신뢰도
- AHS (Account Health Score): 계정 건강도 종합 점수
- PV (Pipeline Velocity): 파이프라인 전진 속도

### 4.3 결과검증 계층 (후행 KPI)

- FGR (Field Growth Rate): 담당 영역 성장
- PI (Prescription Index): 처방 기반 보정 성과
- TRG/SWR: 경영 리포팅용 보조 결과지표 (월/분기 기준)

## 5. 지표 역할 정의 (충돌 방지)

1. HIR/RTR/BCR/PHR는 "행동 개선"을 위한 지표다.
2. NAR/AHS/PV는 "운영 통제"를 위한 지표다.
3. FGR/PI/TRG/SWR는 "결과 검증"을 위한 지표다.
4. 결과지표 단독으로 개인 역량을 판단하지 않는다.

## 6. HIR 중심 운영 설계

### 6.1 HIR 유지 이유

- 입력을 강제하기 어렵고, 모든 행동을 물리적으로 통제할 수 없기 때문에
- 성과기여 행동에 가중치를 둬 "행동 유도"를 만드는 것이 현실적이기 때문

### 6.2 HIR 운영 정의

- 기본식: `HIR = Σ(행동가중치 * 질적점수 * 영향력계수) / 총활동수`
- 목적: 양적 입력이 아니라 유효 행동 비중을 높이는 것

### 6.3 8대 행동지표의 위치

- 8대 행동(Approach, Contact, Visit, Presentation, Question, Need Creation, Demonstration, Follow-up)은 폐지하지 않는다.
- 단, 대시보드 메인 KPI가 아닌 "원인 진단 레이어"로 사용한다.
- 코칭 시 "어떤 행동 축이 약한지"를 세부 처방으로 연결한다.

## 7. 입력 신뢰(무결성) 정책

### 7.1 신뢰도 등급

- Verified: 객관 증빙(시스템 로그/첨부/상호검증) 기반
- Assisted: 일부 근거 + 사용자 보정
- Self-only: 자기기입만 존재

### 7.2 점수 반영 정책

1. Self-only 데이터는 HIR 반영 상한 적용
2. 중복/복붙/추상 메모는 가중치 감점
3. Next Action 불명확 시 PHR/NAR 반영 제한
4. 입력건수 자체는 목표 KPI에서 제외

## 8. 사용자별 운영 시나리오

### 8.1 영업사원

- 오늘의 우선 계정/행동 추천 확인
- 최소 입력으로 활동 기록
- 즉시 피드백: "다음 주 HIR 개선을 위해 필요한 행동"

### 8.2 관리자

- 팀원별 선행지표 갭 확인(HIR/RTR/BCR/PHR)
- 원인코드별 코칭 처방 실행
- 2주 후 NAR/PV/AHS와 결과(FGR/PI) 변화 검증

### 8.3 본부/경영

- 월/분기 성과(FGR/PI/TRG/SWR) 확인
- 선행지표 개선 대비 성과 전환율 검증
- 지역/제품/계정 세그먼트별 편차 관리

## 9. 코칭 폐루프

1. 탐지: 선행지표 이상 감지(HIR/RTR/BCR/PHR)
2. 분해: 8대 행동 축과 계정 컨텍스트로 원인 식별
3. 처방: 2주 실행 액션 부여
4. 이행: NAR/PHR로 이행률 추적
5. 검증: FGR/PI 변화와 상관/시차 분석
6. 학습: 효과 높은 처방을 룰로 승격

## 10. 데이터/시스템 설계 원칙

### 10.1 필수 엔티티

- users, accounts, contacts, activities, outcomes, coaching_cases, pipeline_items, kpi_snapshots, data_reliability_logs

### 10.2 품질 규칙

1. `next_action_date`는 미래 날짜만 허용
2. 설명 텍스트 최소 길이 정책 적용
3. 동일 계정/동일 텍스트 반복은 중복 감점
4. 근거 없는 self-score는 상한 제한
5. 결측은 0이 아닌 `unscored`로 처리

### 10.3 계산 주기

- 선행지표(HIR/RTR/BCR/PHR): 일 배치 + 주 확정
- 운영지표(NAR/AHS/PV): 일 배치 (PV는 이벤트 갱신 병행)
- 결과지표(FGR/PI/TRG/SWR): 일 배치 + 월/분기 확정

## 11. 거버넌스 및 평가 원칙

### 11.1 조직 책임

- SFE PM: 지표 정책/코칭 룰 오너
- 영업운영: 데이터 품질 및 마스터 관리
- 데이터팀: 계산 검증/상관분석/모델 교정

### 11.2 평가 원칙

1. 개인 보상에 입력량 KPI를 직접 반영하지 않음
2. 선행행동 개선과 코칭 이행을 우선 반영
3. 결과지표는 외생변수 보정 후 보조 반영

## 12. 90일 도입 로드맵

### D1~D30 (정렬)

- KPI 계층 확정(선행/운영/결과)
- HIR 및 8대 행동 진단 레이어 복원
- 신뢰도 등급(Verified/Assisted/Self-only) 적용

### D31~D60 (운영)

- 관리자 코칭 루프 본가동
- 팀별 2주 처방 실험
- 목표: NAR/PHR 개선, 저신뢰 입력 비중 감소

### D61~D90 (검증)

- 선행지표와 FGR/PI 시차 상관 검증
- 지역/제품/계정 난이도 보정 반영
- 목표: 코칭군의 유의미한 성과 개선 확인

## 13. Go/No-Go 기준

- 신뢰성: 저신뢰(Self-only) 입력 비중 지속 감소
- 실행성: 코칭 처방 이행률(NAR) 목표 달성
- 효과성: 선행지표 개선군의 FGR/PI 유의 개선
- 수용성: 현장 만족도 및 재사용 의향 개선

## 14. 결론

이 기획안의 본질은 지표 추가가 아니라 운영 철학의 전환이다.

1. CRM 입력을 강제하지 않는다.
2. 좋은 행동을 하게 만드는 선행지표를 중심에 둔다.
3. 후행지표는 성과 검증에 사용한다.
4. 선행행동 -> 운영이행 -> 결과검증의 구조를 반복해 인과를 축적한다.

즉 본 리빌드는 "보고 시스템"이 아니라, "행동 변화 시스템"으로 정의한다.
